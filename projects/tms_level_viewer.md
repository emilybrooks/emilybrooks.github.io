## Motivation
Recently, I've been doing a lot of self study. I didn't have the opportunity to take classes at a university, but a surprising amount of lectures and coursework are available online, even if you aren't a student. One of the courses I was working through has a final project where the assignment was to just make something. They challenge you to "create something that outlives this course."

Something that had interested me for a long time was computer graphics. I was going through some lectures for that as well, but I didn't really have anything concrete to apply it to.

One night I was watching Vinesauce. He was doing a segment where he explored levels from video games using something called noclip.website. This wasn't the first time I'd heard of it, but I had kinda forgotten about it up until now.

All these things were on my mind, so I put two and two together and thought "This might fit the bill".

## Getting Started
The game I decided to add was Tokyo Mirage Sessions ♯FE (TMS♯FE). I absolutely adore it. The gameplay is such an intriguing puzzle to solve I've spent at least 800 hours routing and speedrunning it. It's a chaotic combination of the Fire Emblem series with an Atlus RPG that is practically exploding with color.

So where do I start? I needed to figure out what kind of data I was working with. Games commonly have a "pak" file format to bundle together multiple files for an asset, and TMS♯FE stores it's levels in a single file: model.apak.

A friend made an offhand comment that TMS♯FE used the same model format as other Nintendo developed games. My first thought was to drag our mysterious model.apak into a tool called Switch Toolbox, which supports a smorgasbord of model formats from various Nintendo games. It opened without any issues?? That was easy.

![](/projects/assets/tms_level_viewer/switch_toolbox_file_tree.webp)

A level has multiple models, each stored in a .bfres file. These are generated by NintendoWare for Cafe, a set of proprietary tools Nintendo developed to export assets and render them on Nintendo hardware. The same format was used for Mario Kart 8, Breath of the Wild, and many other games.

Say you have a model you made in Blender, and you want to open it in Maya. You can't just copy and paste it between programs. And Maya can't open your .blend file. 3D models are surprisingly complicated. Every modeling program has their own way of doing things, and they tend to be wildly different from each other.

If you wanted to transfer your model, you'd have to save it to an intermediate format that both programs support. They're all sorts of jank. As an example, one of the most common ones, .fbx, isn't even open source. Other programs had to reverse engineer the format just to add support for it.

In the process of converting from Blender to .fbx to Maya, everything will go wrong. Materials won't display correctly, the scale might be completely wrong, maybe the coordinate system is different and your model is rotated the wrong way. Data is inevitably modified in the process.

In the pursuit of accuracy, noclip.website's philosophy is to read the original 3D graphics data, straight from a game's filesystem. Which means we get to reverse engineer some proprietary file formats, yay.

noclip.website actually has a renderer for Mario Kart 8 levels, so there's some code to parse through .bfres files already. But when I tried to parse a TMS♯FE file, it didn't get very far before failing some assertions.

The .bfres format was used throughout the entire lifespan of both the Wii U and Switch, and there's been a ton of revisions over the years. TMS♯FE started it's life as a Wii U game released in 2015, and was later ported to the Switch in 2020, so I had access to the level models in two different versions: 3.5.0.2 for Wii U and 0.9.0.0 for Switch. But the noclip.website code was designed for version 0.5.0.3. Dang.

While in the end I updated noclip.website's code to support the version I needed, I started out by writing entirely separate code to parse .bfres files from scratch. This was super helpful to get a grasp of everything that was going on, rather than aimlessly debugging the previous code I knew nothing about.

I initially started with Wii U, but I swapped to Switch after Jasper, the owner of noclip.website, suggested that it would be easier. The Wii U processor reads memory in big endian, while modern pc processors read memory in little endian. Using the Wii U files would require converting to little endian, which I could avoid by using Switch files.

I spent many hours staring at bytes in a hex editor, documenting .bfres files as well as other formats used by TMS♯FE. During this project I discovered a really cool program called 010 Editor. You create "binary templates" where you define structs in a c like language, and the program uses these templates to display formatted data.

Since I conceptually understood how animations worked from documenting c_keyframe.c, I wanted to give it a shot. What could go wrong?

Animations are broken down into one dimensional components, like translation on the x axis or rotation around the y axis. To animate one of these components, you store a list of keyframes which mark what the value should be at certain frames of the animation. Storing a keyframe for every frame of the animation would be too expensive, so instead only a few keyframes are stored. For frames without a keyframe, the value is calculated by interpolating between the two surrounding keyframes.

Animal Forest used hermite cubic interpolation, where you store the value and the velocity of the curve for each keyframe. When you interpolate, you input t, a value between 0 and 1 that represents the position between two keyframes. Everything gets plugged into a mathematical mess that's constructed by the four hermite basis functions, and you get the interpolated value.

```
/**
 * @param t Represents the desired position between keyframe 1 and keyframe 2 to interpolate.
 * @param duration The amount of time between keyframe 1 and keyframe 2, in seconds.
 * @param p0 The start point.
 * @param p1 The end point.
 * @param v0 The velocity at p0.
 * @param v1 The velocity at p1.
 * @return Interpolated value.
 */
f32 cKF_HermitCalc(f32 t, f32 duration, f32 p0, f32 p1, f32 v0, f32 v1) {
    // Hermite basis functions
    f32 h3 = 3.0f * SQ(t) - 2.0f * CB(t);
    f32 h2 = -SQ(t) + CB(t);
    f32 h1 = CB(t) - 2.0f * SQ(t) + t;
    f32 h0 = 1.0f - h3;

    return h0 * p0 + h3 * p1 + (h1 * v0 + h2 * v1) * duration;
}
```

According to the Custom Mario Kart 8 Wiki, these should also be hermite curves. They mention that each keyframe has four numbers, for which "It is unclear how to exactly interpret them to form the curve out of it." Oh.

But we're just getting started! The animations in Illusory Daitama only use a single bone per mesh. What about multiple bones per mesh?

Animated meshes have two additional vertex attributes: blend indices and blend weights. These specify up to 4 bones that influence the mesh, and how much influence each bone has. So it should be as simple as writing something like this right?

WorldFromModelMatrix += u_BoneMatrix[int(a_BoneIndices.z)] * a_BoneWeights.z;

My suffering was due to a fundamental misunderstanding of how skinned meshes work. Consider a mesh with only one bone. Its vertices are stored near the origin. Wherever the bone goes, they go. You can think of them in a coordinate space where the bone is the center of the world. Let's call this bone local space.

But when you have multiple bones, which bone do you use for the center? You could pick one, but it doesn't really help. You need a way of getting the vertices into the coordinate space of every bone that influences it. So instead of storing the vertices in relation to a bone, let's create a new coordinate space: the bind pose.

![](/projects/assets/tms_level_viewer/bind_pose.jpg)
If you've ever seen a T pose or an A pose, that's usually what a bind pose looks like.
{: .fs-2 }

But how do we animate this? We need to transform these vertices from the bind pose into each separate bone's coordinate space. We're gonna need some transformation matrices, but we need a whole array of them, one for each bone. These matrices were of course in the model data all along, so I had to hunt them down. I ended up labeling them bone local from bind pose matrices, which is descriptive but a little wordy. I've also seen them referred to as inverse bind pose matrices.

Once we're in bone local space, now we can multiply the bone's transformation matrix and get the proper vertex positions:

![](/projects/assets/tms_level_viewer/barry_pose.jpg)
bind pose space -> bone local space -> world space
{: .fs-2 }

So in short, all I had to do was multiply some matrices together, and my trees were no longer abominations! Shibuya no longer had all the trees and street lamps in the middle of the street, The giant mannequins in Illusory 106 were no longer clipping into walls, and I could successfully render character models!

## Gimmicks
In addition to the world geometry, levels in TMS♯FE contain a bunch of dynamically placed objects, referred to internally as gimmicks.

![](/projects/assets/tms_level_viewer/common_gimmicks.jpg)
breakside, breakwall, warp, and treasurebox gimmicks
{: .fs-2 }

I had no idea where the positions for these were defined, but I did know that TMS♯FE implements a lot of its gameplay code with Lua scripts. These can be decompiled, and they're mostly readable!

The file structure for Lua scripts is kind of a mess, but I located scripts associated with each level in `/Event/field/script_map/`. The first floor of Illusory 106 has a whole zoo of gimmicks, so I started there. Gimmicks are created in GimmickInitializeScript.lua:

`mapGetLayoutPoint()` appears to be responsible for grabbing the position of the mannequin, and it takes swA1_POINT_MANNEQUIN as an argument, which is defined as 1510 inside gimmick_define.lua.

Right after it there appears to be 3 f32s: -208.7, 11.0, and 37.7. These match the mannequin's XYZ coordinates.

    <source src="/projects/assets/tms_level_viewer/daiba_studio_animation.webm" />
</video>

I had to figure out how it worked. These strips of light are uv unwrapped in a straight line, and the texture simply translates horizontally. Surprisingly simple!

![](/projects/assets/tms_level_viewer/illusory_daiba_studio_uv.webp)

What really surprised me was that the only material animations TMS♯FE uses are translation, rotation, and scale. The majority of them are translation, which is fairly trivial to implement. You just add an offset to the UV coordinates.

Scale was a different story. It's so uncommon I had to go through each map with code that would print if there was a scale animation. The first one that came up was in Illusory Shibuya Central Square. There's a large mural of Maiko from the cover of My Complex. It starts at 1x scale, and takes about 30 seconds to zoom in to 2x scale.

<a href="https://noclip.website/#TokyoMirageSessionsSharpFE/d004_05;ShareData=AP6OMUlX~&7]Lc~8mkVw9z_pPQ4$dKUn]}Y9FX,)Vx5XvTc)X5T=CQ?Udy2iVt">
![](/projects/assets/tms_level_viewer/maiko_wall_ingame.jpg)
</a>

If translation means adding, scaling should just mean multiplying right? Not quite. The question we have to ask is "where are we scaling from?". Scaling happens in relationship to the origin, but that wasn't giving the correct result here:

<a href="http://noclip.website/#TokyoMirageSessionsSharpFE/d004_05;ShareData=ATHoQUm5{y8TziH8[uF?+ileWP[I;cUtJSP8{FtCV6*OTT6-xsT;CX}Ui+Eq+C">
![](/projects/assets/tms_level_viewer/maiko_wall_wrong1.jpg)
</a>

In order to scale from a different point, you first translate so that point is the new origin. then scale. then translate back the opposite amount. This animation has the texture scale from the middle of the mural, so maybe I should translate (-0.5, -0.5) and scale? but that didn't work either:

<a href="http://noclip.website/#TokyoMirageSessionsSharpFE/d004_05;ShareData=ATHoQUm5{y8TziH8[uF?+ileWP[I;cUtJSP8{FtCV6*OTT6-xsT;CX}Ui+Eq+C">
![](/projects/assets/tms_level_viewer/maiko_wall_wrong2.jpg)
</a>

At this point I resorted to looking through the Mario Kart 8 renderer. It uses a matrix to transform the uv coordinates, and there were actually 3 different flavors for Maya, 3DSMax, and XSI. The Maya one magically solved all my problems:

```
function calcTexMtx_Maya(dst: mat4, scaleS: number, scaleT: number, rotation: number, translationS: number, translationT: number): void {
    const theta = rotation * MathConstants.DEG_TO_RAD;
    const sinR = Math.sin(theta);
    const cosR = Math.cos(theta);

    mat4.identity(dst);

    dst[0]  = scaleS *  cosR;
    dst[4]  = scaleS *  sinR;
    dst[12] = scaleS * ((-0.5 * cosR) - (0.5 * sinR - 0.5) - translationS);

    dst[1]  = scaleT * -sinR;
    dst[5]  = scaleT *  cosR;
    dst[13] = scaleT * ((-0.5 * cosR) + (0.5 * sinR - 0.5) + translationT) + 1.0;
}
```

At this point I could have just moved on, but I didn't understand what the heck the math was doing. It didn't sit right with me to just blindly copy some code without knowing how it worked. 

The rotation was making everything complicated, so I removed it from the equation by setting theta to 0. CosR evaluates to 1, and SinR evaluates to 0:

```
dst[0]  = scaleS *  1;
dst[4]  = scaleS *  0;
dst[12] = scaleS * ((-0.5 * 1) - (0.5 * 0 - 0.5) - translationS);

dst[1]  = scaleT * -0;
dst[5]  = scaleT *  1;
dst[13] = scaleT * ((-0.5 * 1) + (0.5 * 0 - 0.5) + translationT) + 1.0;
```

Which can be simplified to:

```
dst[0]  = scaleS;
dst[4]  = 0;
dst[12] = scaleS * -translationS;

dst[1]  = scaleT * -0;
dst[5]  = scaleT *  1;
dst[13] = scaleT * (-1 + translationT) + 1.0;
```

If you set the translation to (0, 0), you're left with this:

```
dst[0]  = scaleS;
dst[4]  = 0;
dst[12] = 0;

dst[1]  = 0;
dst[5]  = scaleT;
dst[13] = -scaleT + 1.0;
```

The important line is the bottom one. This looks like a normal scale matrix, but it was multiplied with a matrix that translates -1 on the vertical axis, then translated back 1.

![](/projects/assets/tms_level_viewer/scale_matrix.webp)

The vertex buffers store UV coordinates with the top left as the origin. If you translate (0, -1) that would bring the bottom left corner to the origin. Which means we should be scaling around the bottom left, not the center.

Funnily enough, the texture in question actually hints towards this conclusion:

![](/projects/assets/tms_level_viewer/d004_05_obj1_picture_maiko_glow01.jpg)

In order to get the texture to scale from the center, the artist chopped it up into quadrants and rearranged it so that the center was in the bottom left.

This is how the mural is UV mapped:

![](/projects/assets/tms_level_viewer/maiko_uv1.jpg)

In order to scale it correctly, you first it translate -1 on the vertical axis:

![](/projects/assets/tms_level_viewer/maiko_uv2.jpg)

Then scale by 0.5 (which scales around the origin in the top left):

![](/projects/assets/tms_level_viewer/maiko_uv3.jpg)

And then translate back 1 on the vertical axis to undo the first translation:

![](/projects/assets/tms_level_viewer/maiko_uv4.jpg)
